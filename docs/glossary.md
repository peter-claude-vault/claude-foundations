# Glossary

Every term used in claude-stem documentation, defined in one sentence. Where a term has a longer reference doc, follow the link.

## Core terms

- **Claude Code** — Anthropic's CLI for Claude. You type `claude` in a terminal and get a chat session with the model that can run shell commands, read and edit files, and call tools. It reads `~/.claude/` for user configuration.

- **Vault** — your folder of markdown notes. The system uses [Obsidian](https://obsidian.md)'s vault concept: a directory that contains `.md` files plus a few subdirectories (Inbox, Logs, Plans). You don't need Obsidian itself; any editor works.

- **Manifest** — a JSON file at `~/.claude/user-manifest.json` that holds your name, vault path, role, preferences, and dozens of other configuration fields. The single most-read file in the system. Think "config file, but generated for you, structured, and validated against a schema." Several other JSON files in the system are also called manifests (`librarian-manifest.json`, `foundation-manifest.json`, per-plan `manifest.json`); when this docs say "the manifest" without qualification, it means `user-manifest.json`.

- **Hook** — a shell command Claude Code runs at lifecycle events. The events the system wires include `PreToolUse` (before a file write), `SessionStart`, `Stop`, `PreCompact`, `SessionEnd`, plus a few others. See [`hooks/README.md`](../hooks/README.md).

- **Skill** — a slash command you type in Claude Code, like `/onboard` or `/librarian`. Each lives in a directory under `~/.claude/skills/` with a `SKILL.md` body Claude reads at invocation time.

- **Slash command** — a command typed in the Claude Code prompt starting with `/`. Maps 1:1 to skills.

- **Frontmatter** — the YAML block at the top of a markdown file, between two `---` delimiters. Carries structured metadata (`type`, `tags`, `created`, etc).

## File types and structure

- **System Backlog** — a single markdown file at the root of your vault (`System Backlog.md`) that tracks the system-improvement projects you've decided to take on. Modified by `/backlog-triage` and `/backlog-research`.

- **Plans** — a directory tree (`~/.claude-plans/`) where multi-step initiatives are tracked. Each plan is a numbered directory with a `spec.md`, a `tasks.md`, and a `manifest.json`. The `Plans/` symlink in your vault points here.

- **Plan tree** — colloquial for `~/.claude-plans/`.

- **Routing Decision Tree (RDT)** — the section of your vault `CLAUDE.md` that tells Claude where to put a new file based on what it is (meeting note → here, project doc → there, reference → elsewhere). Generated by Section F's vault `CLAUDE.md` surface.

## Archetypes and personalization

- **Archetype** — a pre-defined category of knowledge worker (consultant, researcher, developer, educator, manager, generalist, writer). Used to seed defaults during onboarding (canonical file types, tag prefixes, architect research topics). Detected from your Section B/C transcripts via a deterministic keyword scorer; you can override.

- **Engagement** — consultant terminology for "client project". A first-class concept when your archetype is consultant. The default `vault.top_level_folder` is `Engagements`. Other archetypes use different top-level folder names.

- **Engagements-based** — a value of `vault.organizational_method`. Indicates the vault's top-level partition is by client / project rather than (e.g.) by topic or by date.

- **Personalization tier** — Universal / Combined / Personal. Universal artifacts ship generic for every user. Combined artifacts ship a generic skeleton with per-user slot values. Personal artifacts are composed entirely from your interview answers. See [`personalization-model.md`](personalization-model.md).

- **Personalization surface** — one of the seven distinct files / configurations onboarding's Section F writes to with LLM help. Examples: claude-home `CLAUDE.md`, vault `CLAUDE.md`, memory seeds, tag prefixes, doc-dependencies registry. Each surface has its own LLM-or-deterministic mode.

- **Section F** — the optional auto-authoring phase that runs after the standard 5-section interview (Sections A–E). Dispatches the seven personalization surfaces and (with `--seed-content`) the four-stage infer-vault chain.

## Onboarding internals

- **Section A / B / C / D / E** — the five interview sections. A: identity confirmation. B: who you are and what you do. C: your knowledge system. D: trust, privacy, automation. E: final binary toggles.

- **Q-ID** — a stable identifier for an interview question (e.g. `B-3`, `D-2`, `A-CB1`). Maps to a manifest field via `q-field-map.json`. Stable across releases.

- **q-field-map** — the JSON file at `onboarding/q-field-map.json` that defines the Q-ID → manifest-field mapping. Code iterates the map's keys; never enumerates Q-IDs inline.

- **Confidence gate** — the rule applied to every LLM-extracted field. ≥ 0.85: populate silently. 0.5–0.85: yellow-confirm in the summary. < 0.5: surface as missing; trigger one surgical follow-up; re-extract once. Block section exit if a required field is still < 0.5 after the follow-up.

- **Opt-out surface** — a place in the interview where you can decline to share or capture something (organization, people, vault, sensitive content acknowledgement, etc.). There are ten of them.

## Infer-vault chain

- **Infer-vault chain** — a four-stage pipeline that takes a corpus of seed content (your existing notes), clusters by similarity, proposes a taxonomy via LLM, renders a markdown plan you can edit, and runs a 3-step gate before any vault writes. The four stages: cluster → propose-taxonomy → import-plan → review-gate.

- **IR (intermediate representation)** — the JSONL produced by Stage 1 (`ir-builder.sh`). One record per source file, with content plus metadata. Consumed by the cluster stage.

- **Stub mode** — a deterministic fallback for stages that would normally call an LLM. Used when no API key is available or when `--llm-mode stub` is forced. Reproducible.

- **Three-step gate** — the apply / edit / skip / abort interaction for any auto-authored artifact. You see the proposed write, optionally edit it, and choose to commit or not. Library at `onboarding/lib/three-step-gate.sh`.

## Runtime guards

- **R-NN rule** — a numbered policy rule enforced by `pre-write-guard.sh` or surfaced as a librarian finding. The numbering exists because rules appear by number in audit logs; renumbering breaks log archaeology. The thirteen active R-rules are listed in [`hooks/RULES.md`](../hooks/RULES.md).

- **Output Contract** — a section every skill that writes to the user filesystem must include. Declares files written, schema types, pre-write validation steps, and failure mode. Enforced by code review.

- **Block-and-log** — the canonical failure mode. On any validation, parse, or IO failure: roll back staged tempfiles, append a structured log entry, exit non-zero. Never "write and hope".

- **Drift sweep** — a librarian capability that scans the vault for content that has evolved away from declared spec (frontmatter that no longer matches the schema, files placed where the routing tree wouldn't put them).

- **Cascade** — the doc-dependency mechanism. When you write to a registered "primary" file, you see an advisory listing the "mirror" files to update. Configured in `hooks/config/doc-dependencies.json`.

## Cron and orchestration

- **launchd plist** — a macOS scheduled-job descriptor. Each cron job in the system has a plist template at `templates/launchd/`. Rendered to `~/Library/LaunchAgents/` and bootstrapped via `launchctl bootstrap`.

- **Idle watchdog** — a wrapper around long-running cron jobs that kills the process if it produces no log output for N minutes (default 180s). Handles cold-wake hangs after macOS sleep.

- **Orchestration manifest** — `orchestration.json`. Holds the cron job definitions (schedule, log path, idle-watchdog seconds, model, budget).

## Connectors

- **Connector** — a wired-up MCP server that pulls data from a third-party service into your vault on a schedule. Granola is the reference example.

- **MCP** — Model Context Protocol. Anthropic's standard for tools and integrations.

- **Pipeline template** — a JSON file at `connectors/templates/<name>.json` that declares (as data, not code) the full flow for one connector: query, processor skill, target vault path, schedule, failure mode.

## Plan / project terms

- **`parent_plan`** — a frontmatter field on sub-plan files. Points to the slug of the top-level plan the file belongs to. Used by the librarian to resolve plan dependencies.

- **NN- prefix** — every plan directory has a numeric prefix in creation order (`38-claude-foundations-onboarding-engine`, `71-claude-foundations-engine-v2`). The pre-write guard refuses to create a plan without one.

- **Status header** — every plan's `spec.md` carries `**Status:**` near the top; alternatively the plan's `manifest.json` has a `status` field. Used by `/librarian plan-index` to group plans.

## Internal naming you might run into

- **G1, G1-pre, G1-main** — install-script guards. G1-main is the existing-`~/.claude/` clobber refusal; G1-pre is the `$CLAUDE_HOME` env-var presence check.

- **`I-UNDERSTAND-APRIL-13` sentinel** — the literal confirmation phrase the installer demands before overwriting an existing `~/.claude/`. Named after the date of the install-corruption incident; documented at [`install-corruption-incident.md`](install-corruption-incident.md).

- **CFF (Cross-File Findings)** — internal terminology for a librarian finding that spans multiple files (e.g. provides-canonicality drift). You'll see the codes in finding output.
